# Setup Guide

## Requirements

- Docker + Docker Compose v2
- `curl` and `jq` (for the CLI demo script)
- 8 GB RAM minimum (SonarQube is memory-hungry)

**One-time host setting required by SonarQube's embedded Elasticsearch:**

```bash
sudo sysctl -w vm.max_map_count=262144
```

To make it permanent, add `vm.max_map_count=262144` to `/etc/sysctl.conf`.

---

## First-Time Setup

### 1. Copy environment file

```bash
cp .env.example .env
```

### 2. Start SonarQube alone first

```bash
docker compose up -d sonarqube
```

Wait for it to be healthy — this takes about 60–90 seconds on first boot.

```bash
docker compose ps   # wait until sonarqube shows "healthy"
```

### 3. Initialize SonarQube (run once only)

This script changes the default admin password and generates an analysis token:

```bash
bash scripts/init-sonar.sh
```

When prompted, enter a password of at least 12 characters. The script will print two lines to add to your `.env`:

```
SONAR_TOKEN=sqa_...
SONAR_ADMIN_PASSWORD=your_password
```

Copy those into `.env`. Also set a database password:

```
POSTGRES_PASSWORD=choose_a_strong_password
```

### 4. Build and start all services

```bash
docker compose up -d --build
```

This starts six services: `redis`, `sonarqube`, `api`, `worker`, `postgres`, `web`.

### 5. Verify everything is running

```bash
docker compose ps
```

All services should show as `running` or `healthy`. The web UI is at **http://localhost:8080**.

## Running an Analysis

### Via the web UI

Open **http://localhost:8080**, paste a GitHub repository URL, and click **Analyze Repository**. Progress updates live via SSE. Results show a 5×5 risk matrix, phase scores, Dev↔Ops correlations, and a sortable findings table.

### Via the CLI script (terminal)

```bash
bash scripts/run.sh https://github.com/owner/repo
```

Polls the API every second and prints the full JSON result when complete.

### Via the API directly

```bash
# Submit
curl -s -X POST http://localhost:3000/analyze \
  -H "Content-Type: application/json" \
  -d '{"repoUrl": "https://github.com/owner/repo"}' | jq

# Poll by jobId
curl -s http://localhost:3000/jobs/<jobId> | jq

# Stream progress (SSE)
curl -N http://localhost:3000/jobs/<jobId>/stream
```

For private repositories, include a GitHub personal access token:

```bash
curl -s -X POST http://localhost:3000/analyze \
  -H "Content-Type: application/json" \
  -d '{"repoUrl": "https://github.com/owner/private-repo", "githubToken": "ghp_..."}' | jq
```

## Subsequent Starts

Once everything is initialized, starting and stopping is straightforward:

```bash
# Start
docker compose up -d

# Stop (preserves all data volumes)
docker compose down

# Stop and wipe all data (fresh start)
docker compose down -v
```

## Configuration

All settings are in `.env`. Key variables:

| Variable | Default | Description |
|---|---|---|
| `SONAR_TOKEN` | — | Generated by `init-sonar.sh`. Required. |
| `SONAR_ADMIN_PASSWORD` | — | Set during init. Required. |
| `POSTGRES_PASSWORD` | `changeme` | Password for the analysis history database. |
| `WORKER_CONCURRENCY` | `1` | Concurrent analysis jobs. Keep at 1 unless the host has >16 GB RAM (each job runs 5 scanners in parallel). |
| `GIT_CLONE_DEPTH` | `50` | Git clone depth. Deeper = better git hygiene metrics, slower clone. |
| `SKIP_TRIVY` | `false` | Set `true` to skip CVE and IaC scanning. |
| `SKIP_GITLEAKS` | `false` | Set `true` to skip secret detection. |
| `SKIP_HADOLINT` | `false` | Set `true` to skip Dockerfile linting. |
| `SKIP_CHECKOV` | `false` | Set `true` to skip IaC misconfiguration scanning. |
| `RISK_MAPPINGS_PATH` | built-in | Path to a custom `risk-mappings.json` to override impact/likelihood scores. |

### Customizing the risk matrix scores

Edit `packages/shared/config/risk-mappings.json` to change how each tool's severity maps to `{impact, likelihood}` on the 1–5 scale. The file is loaded at worker startup — rebuild the worker container to apply changes:

```bash
docker compose build worker && docker compose up -d worker
```

Or point to an external file without rebuilding:

```bash
RISK_MAPPINGS_PATH=/path/to/my-mappings.json docker compose up -d worker
```

## Architecture at a Glance

```
Browser → web (Nginx :8080)
              ↓ /api/* proxy
           api (Fastify :3000)
              ↓ BullMQ job
           redis ← worker
                      ↓ parallel
              sonarqube   trivy / gitleaks / hadolint / checkov
                      ↓
                 postgres (history)
```

- **Analysis time:** 3–8 minutes per repo (SonarQube dominates; ops tools finish in under 2 minutes).
- **Results TTL:** 24 hours in Redis. Permanent storage in PostgreSQL (once the P1 persistence feature is wired up).
- **Ports exposed:** `8080` (web UI), `3000` (API), `9000` (SonarQube dashboard).

## Troubleshooting

**SonarQube never becomes healthy**
- Check `docker compose logs sonarqube`. The most common cause is `vm.max_map_count` being too low. Run `sysctl -w vm.max_map_count=262144` on the Docker host.

**Worker exits immediately**
- `SONAR_TOKEN` is missing or invalid. Re-run `scripts/init-sonar.sh` and update `.env`.

**Trivy/Gitleaks/Checkov not producing results**
- The tools download databases or configs on first use inside the container. Check `docker compose logs worker` for download progress. First job on a fresh container may be slower.

**"Repository is private or does not exist"**
- Pass a `githubToken` with `repo` scope in the request body or web UI token field.

**Port 8080 already in use**
- Change the host port in `docker-compose.yml`: `"8081:80"` under the `web` service.
